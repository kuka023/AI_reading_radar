# AI 读书雷达 — 设计文档

面向 AI 从业者的英文技术书雷达。竞品是 ThoughtWorks 读书雷达。

---

## 0. 为什么做 / 差异点在哪

ThoughtWorks 读书雷达（2013 / 2016 / 2019 / 4.0-2022）本质是**一群专家每三年拍一次板的静态海报**：象限是知识领域，环是难度（入门/进阶/高级），符号区分「强烈推荐」和「推荐」。它值钱的地方不是雷达图，而是有品味的人愿意做取舍。

它有三个结构性做不到的事，就是我们的空间：

1. **它不针对具体的人。** 一张图给所有程序员。
2. **它不更新。** 三年一版，中间世界变了。
3. **它从不说「别读」。** 只有推荐和强烈推荐。

第 3 点在 AI 领域尤其致命，也是本产品的核心命题：

> **AI 是唯一一个「读书雷达」真正需要 Hold 环的领域。**
> 一本讲 TF 1.x 的实战书、一本 pre-transformer 的 NLP 书，今天不是"进阶读物"，是**负资产**——读了会学到错误的心智模型。
> "这本别读了，它讲的范式已经过时" 是这个领域最稀缺的判断，也是我们唯一的护城河。

如果最后做出来的东西不敢说 Hold，那我们只是做了一个用 AI 生成的书单，没有意义。

## 1. 已定的决策

| 维度 | 决策 |
|---|---|
| 环的语义 | 采纳建议（立场），非难度 |
| AI 职责 | 定位判断、个性化重排、理由与路径生成 |
| 不做 | 自动策展入库（书目库先人工策展） |
| 受众 | AI 从业者（研究、工程、应用、产品） |
| 书目范围 | 英文技术书为主 |

**书目范围选英文，解决了最大的工程风险。** 中文书元数据是个泥潭（豆瓣无开放 API，Google Books / Open Library 中文覆盖差），英文技术书则可以直接从 Open Library / Google Books 拿到干净的 ISBN、出版年、版次。这让我们能把精力花在判断逻辑上，而不是刮数据。

### 待确认（已填推荐值，可改）

- **象限 = 知识层次**：基础与理论 / 模型与算法 / 工程与系统 / 产品与应用。
  选这个而不是"按岗位切"，因为岗位已经是用户画像了，再当象限会让雷达退化成四份独立书单。
  层次切法还有个附带好处：**越靠上层，知识半衰期越短**，雷达图上会自然浮现出"哪里在腐烂"。
- **blip 只放书**（v1）。数据 schema 预留 `type` 字段，为将来放经典论文/课程留门。
- **第一版做可交互原型**：静态站点 + 人工策展的书目 JSON + 一次 LLM 调用完成个性化。先验证形式成不成立，再谈后端和账号。

## 2. 环的语义（产品的灵魂，必须先咬死）

技术雷达的 Adopt / Trial / Assess / Hold 迁移到书上：

| 环 | 对书的含义 | 判断主体 |
|---|---|---|
| **Adopt — 现在就读** | 地基性的。做这行不读会有明显缺口。 | 因人而异 |
| **Trial — 值得投入** | 有真价值，但不普适。只对特定角色/目标值得花那几十个小时。 | 因人而异 |
| **Assess — 知道即可** | 不必通读。参考书、或理念重要但可按需查阅。 | 因人而异 |
| **Hold — 不必读了** | 过时、被取代、名过其实。 | **对所有人一致** |

### 关键设计：Hold 是客观的，内三环是个性化的

一本书**是不是已经死了**，是书自身的属性（它讲的框架换了、范式被取代了）——对谁都一样。
一本书**该不该由你现在读**，才因人而异。

这样两层判断自然压进一个环，语义不打架：

```
Hold      ← 书的客观属性（衰变判断）
Assess ┐
Trial  ├─ 书 × 人 的关系（个性化判断）
Adopt  ┘
```

工程上就是：先跑**衰变模型**筛掉死书（全局，可缓存），再跑**个性化模型**排剩下的（按用户，实时）。

## 2.5 难度：是符号，不是环

**「经典」「难不难」「该不该你现在读」是三个正交的维度。** 把它们搅在一起，雷达就废了。

曾经考虑过「环 = 难度，内难外易」。它必须被否掉，原因很硬：

**难度轴上没有「别读」这个位置。** Hold 环无处可放 → 衰变引擎无事可做 → 护城河归零 → 做出来的就是 ThoughtWorks 读书雷达（他们的方案恰恰是「象限=领域，环=难度」）。

更根本的是：**TW 为什么必须把难度做成环？因为他们没法个性化。** 一张静态海报给所有程序员，唯一能做的就是把难度暴露出来，让读者自己对号入座。**难度环是「无法个性化」的补丁。** 我们没有这个约束——个性化层已经有 `math_tolerance` 和 `read`：

> PRML 对数学承受度低的人落在 **Assess**，对研究员落在 **Adopt**。
> **难度不是环，难度是决定你落在哪个环的输入。**

抄难度环，等于把自己的优势换成对方的补丁。

### 那「由浅入深」放哪

两个地方，都不占用环：

1. **blip 符号** —— `● 入门　◆ 进阶　▲ 硬核`（技术雷达本来就这么干：环表达立场，符号表达别的维度）
2. **阅读路径**（`path` 字段）—— **「由浅入深」本质是一条路径，不是一个圈层。** 路径能表达顺序和依赖（读完 A 才读得动 B），圈层不能；而且路径可以因人而异。

### 难度作为硬约束

`math_tolerance = low` 的人，`hardcore` 的书**禁止**进 Adopt——那只会让他在第三章放弃，然后再也不回来。落 Assess，理由写「等你需要时再来查」。这条规则写死在 personalize.py 的 system prompt 里。

## 3. 衰变模型 —— 这是最该做扎实的一环

**不要让 LLM 凭印象判断一本书过没过时。** 它会给出一个听起来很有道理但完全不可靠的答案。

一本 AI 书是否过时，可以从结构化事实**机械地推出来**。给每本书标注：

```jsonc
{
  "isbn": "9781492032649",
  "title": "Hands-On Machine Learning with Scikit-Learn, Keras & TensorFlow",
  "published": 2019,
  "edition": 2,
  "latest_edition": { "edition": 3, "published": 2022 },   // 有新版 → 旧版直接 Hold

  // 衰变信号 —— 人工/半自动标注，是判断的证据基础
  "decay": {
    "paradigm": "pre-transformer",      // pre-DL | DL | pre-transformer | transformer | LLM | agentic
    "pinned_apis": ["tensorflow-1.x", "keras-legacy"],  // 绑死了哪些具体 API/版本
    "superseded_by": ["9781098125974"], // 被哪本书取代
    "content_type": "hands-on"          // hands-on 衰变最快；theory/math 最慢
  }
}
```

### ⚠️ 原型抓到的真 bug：范式阶梯是个错误的抽象

第一版规则里有这么一条：「`paradigm` 落后当前 2 代以上 **且** 是动手书 → Hold」。

跑起来它把 **Hands-On ML 第 3 版（2022）** 和 **Chollet 的 Deep Learning with Python** 全判死了。这两本都是当前最好的书。**12 本里误杀 2 本 —— Hold 环 17% 的误判率，产品第一天就没人信了。**

错在抽象本身：我把 AI 建模成一条线性阶梯（`pre-dl → dl → transformer → llm → agentic`），拿单一的「当前范式」去量每一本书。但**这个领域不是线性演进的**：agent 没有淘汰监督学习，transformer 没有淘汰梯度提升树在表格数据上的地位。它们是不同的层，不是先后的代。任何「离最新范式有多远」的度量，都会机械地处死一切不讲最新东西的书。

**修正后只保留三条判死规则，每一条都建立在能被独立核实的硬事实上：**

| 规则 | 触发条件 | 为什么可信 |
|---|---|---|
| `superseded_edition` | 存在同书更新版次 | 硬事实，零争议 |
| `dead_api` | 绑定已废弃的框架大版本（TF 1.x、NLTK…） | 可核实，零争议 |
| `superseded_technique` | **策展人明确判定**它教的技术路线已被取代（必须填理由） | 人的判断，不可能算出来 —— 老实承认 |

另加一条护栏：`content_type == "theory"` 的书永不因范式判死（数学不会过时）。

修正后 12 本里判死 2 本，**两本都无可辩驳**：HOML 第 2 版（被自己的第 3 版取代）、NLTK 那本（绑死在已被 transformer 全面取代的工具链上）。零误杀。

**教训：不要从「出版年 + 范式」推断死活。那是幻觉的另一种形式 —— 只不过幻觉的主体从 LLM 换成了一条拍脑袋的规则。** 判死必须落在能指着说「因为这个」的事实上，因为 Hold 是要被用户申诉的。

LLM 只负责**把这些事实写成一句人话理由**，不负责下判断。判断来自规则和标注，措辞来自模型。

> 这一节是整个产品最有 defensibility 的地方 —— 而且它**完全不需要 AI**。竞品做不到，是因为他们三年才开一次会。

## 4. 数据地基 与 防幻觉

**绝对不能让模型自由生成书名。** LLM 推荐书会稳定地编造不存在的书、张冠李戴作者、推荐早已绝版的版本。

铁律：

1. **雷达上每个 blip 必须是书目库里的一条真实记录**，通过 ISBN 锚定。模型只能从库里**检索和排序**，不能凭空产出书名。
2. 书目库 v1 **人工策展**，目标 60–100 本覆盖四象限。这个规模一个人一周能做完，且质量远高于任何自动抓取。
3. 元数据从 Open Library API 拉（免费、无 key、有 ISBN/年份/版次）。Google Books 作为补充。
4. 模型的每次输出都做**校验**：返回的 ISBN 必须在库里，不在就丢弃重试。

## 5. 用户画像（个性化的输入）

个性化重排的输入。刻意做小——填写成本越高，冷启动越难。

```jsonc
{
  "role": "applied-llm-dev",     // researcher | ml-engineer | applied-llm-dev | ai-pm | data-engineer
  "goal": "六个月内能独立设计并上线一个 agent 系统",  // 自由文本，最强的个性化信号
  "math_tolerance": "medium",    // low | medium | high —— AI 书最大的分野，决定 Adopt 里放不放 PRML 这类
  "read": ["9781492032649"],     // 已读书 ISBN，比任何自评都准
  "time_budget": "2h/week"
}
```

`goal` 是自由文本，是最有价值的信号，也是 LLM 真正该发挥作用的地方。
`read` 是最可靠的水平推断依据——比让用户自评"我是中级还是高级"准得多。

## 6. 时间维度 —— 雷达的灵魂是「移动」

技术雷达每一期最好看的部分不是那张图，是**变迁**：某个东西从 Assess 升到 Trial、或掉进 Hold。

**没有版本和变迁，雷达只是一个花哨的饼图。**

所以从第一版就要有版本概念，哪怕只是两个快照：

- 每本书记录 `ring_history: [{ version, ring }]`
- 图上用符号标出本期移动的书（↑ 上升 / ↓ 下降 / ● 新入 / ▲ 掉入 Hold）
- 变迁列表本身就是一篇可读的内容（"本季有 3 本书掉进了 Hold"）

个人雷达还有第二种移动：**用户读完一本书后，雷达重排**。读完地基书 → 进阶书从 Assess 升到 Adopt。这个反馈闭环是让用户回来的唯一理由。

## 7. 已知的坑

**冷启动。** 新用户没画像时第一张雷达怎么生成？方案：给 4–5 个预设角色模板（"我是做 LLM 应用的"），一键出图，之后再让他微调。不要一上来让人填表。

**blip 密度。** 一个象限超过 8 本，图就没法看了。英文书名很长（"Designing Machine Learning Systems" 这种）。必须限制每象限每环的 blip 数，或者做悬停/点击展开。这是个真实的视觉约束，会反过来逼迫策展做取舍——这是好事。

**只有书可能不够。** AI 领域最前沿的东西在论文和博客里，不在书里。v1 接受这个局限，但 schema 里留 `type` 字段。

**成功怎么衡量。** 不定义清楚，这就是个玩具。候选指标：用户是否回来更新已读书（说明雷达真的在指导他）> 生成雷达的次数 > 页面停留时长。

## 7.5 分阶段：先不上 AI

**成本不是理由。** 实测个性化一次调用约 1,200 tokens 输入 / 2,000 输出：

| 模型 | 单次 | 1 万张雷达 |
|---|---|---|
| Opus 4.8 | $0.056 | $559 |
| Sonnet 5 | $0.034 | $336 |
| Haiku 4.5 | $0.011 | $112 |

这个数字小到不该影响任何架构决策。

**真正的理由是：AI 现在没有燃料。** 个性化引擎的输入质量决定输出质量。现在输入是一个 12 本的书目库和一个凭空编的画像 —— AI 排出来的东西再漂亮也是空转。

而「官方策展 + 社区推荐打分」恰恰是在造那个燃料：

- **官方策展** → 可信书目库。这是防幻觉的锚（模型只能从库内检索）。没有它，AI 必然编书。
- **社区打分** → ground truth。这是个性化的校准信号。没有它，AI 的「因人而异」就是凭先验瞎猜。

### 阶段一（无 AI，可发布）

官方策展书目库 + 规则衰变引擎 + 社区打分。
**这已经是一个能发布、且已经超过 ThoughtWorks 的产品** —— 因为它有 Hold 环，而且它持续更新。护城河（衰变引擎）在这一阶段就已经建成。

### 阶段二（AI 加入）

等书目库和打分矩阵有了真实数据，AI 再进来做它真正不可替代的两件事：

1. **冷启动预测** —— 新书还没有任何打分数据时，先给它一个位置。
2. **翻译成人话** —— 把「角色 × 目标 × 完读率 × 有用度」的矩阵写成一句能读的推荐理由。

## 7.6 打分怎么设计（这里有个坑）

**打分的经典陷阱是它回答了错的问题。**

「这本书好不好」是豆瓣评分 —— 全世界都有，而且有严重的马太效应：名气大的书分高。PRML 能打 9.5 分，但 90% 的人读不完它。这个分数对「我该不该现在读」毫无信息量。

雷达要的不是「好不好」，是「**该不该你现在读**」。所以问题必须这么问：

| 别问 | 要问 | 为什么 |
|---|---|---|
| 打几分？ | **你读完了吗？** | 完读率比评分诚实得多 —— 它骗不了人 |
| 推荐吗？ | **读完后对你当前的工作有实际帮助吗？** | 这才是雷达环的语义 |
| （不问） | **你是什么角色？当时的目标是什么？** | 这一条让打分可以被切片 |

这样收上来的**不是一个平均分，而是一个「角色 × 目标 × 完读率 × 有用度」的矩阵**。

这个矩阵有两个好处：**它本身就能直接排序，不需要 AI**；而它同时又正好是阶段二个性化引擎需要的训练信号。所以先做打分不是「推迟 AI」，是「给 AI 攒燃料」。

## 8. v1 实施路径

### 已建成（原型跑通）

```
✅ 书目库          data/books.json —— 12 本，含 decay 标注。唯一的事实来源
✅ 元数据校验      src/metadata.py —— 逐条对回 Open Library，防幻觉第一道闸门
✅ 衰变引擎        src/decay.py —— 纯规则，零 LLM。12 本判死 2 本，零误杀
🔨 个性化引擎      src/personalize.py —— 已写好，含强制 ISBN 校验；等书目库和打分数据
🔨 雷达渲染        src/radar.py —— 文本版
```

**校验脚本第一次跑就抓到了真问题**：`/api/books` 端点覆盖有缺口，会把真实存在的书（Hands-On ML 第 3 版）误判成假书。必须用 `/isbn/{isbn}.json` 这个权威端点做闸门。

### 阶段一待办（无 AI）

1. **扩书目库到 60–100 本** —— 一个人一周能做完，质量远高于任何自动抓取
2. **建打分回收** —— 按 7.6 的问法，收「角色 × 目标 × 完读率 × 有用度」矩阵
3. **雷达可视化** —— 四象限四环，参考 ThoughtWorks 开源的 build-your-own-radar
4. **版本与变迁** —— 两版之间做 diff，标出移动（雷达的灵魂）

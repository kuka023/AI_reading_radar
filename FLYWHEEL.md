# 功能规划与数据飞轮

如何让用户进来使用，并让雷达随使用自进化。

---

## 0. 现状诊断：为什么现在没有飞轮

当前产品是**只读的**。用户选角色、看雷达、点书、离开。整个过程不产生任何沉淀，也不改进任何东西。

这不是功能少的问题，是**结构问题**：使用行为没有被转化为数据。飞轮的定义是——用户的使用本身产生数据，数据反过来改进推荐，更准的推荐吸引更多用户，更多用户产生更多数据。这个闭环现在一处都没接上。

同时必须承认一个约束：**排环引擎目前是一张人工先验表**（`placement.py` 里 `researcher → foundations = 10` 这类权重是手写的猜测）。它现在能跑，但它的"个性化"是我拍的，不是数据证明的。飞轮要解决的正是这件事：**用真实使用数据替换先验猜测。**

---

## 1. 飞轮的闭环

```
   官方策展（种子燃料，不依赖用户）
            │
            ▼
   可信的雷达 ——— 有 Hold 环，敢说「别读」
            │
            ▼
   吸引用户 ←──────────────────────────┐
            │                          │
            ▼                          │ 转速加快
   使用 → 产生信号                      │
   （选角色·点击·跳转·书架·完读·        │
     有用度·申诉·提名）                 │
            │                          │
            ▼                          │
   信号校准三大引擎                     │
   （排环 affinity · 衰变规则 · 书目库） │
            │                          │
            ▼                          │
   更准的雷达 ────────────────────────┘
```

**这个闭环有一个先有鸡还是先有蛋的悖论**：飞轮需要数据才能转，但没数据时产品不够好，吸引不来用户。破解见第 4 节。

---

## 2. 信号的摩擦阶梯（这是整份规划最关键的部分）

飞轮的燃料是用户信号。信号按**采集摩擦**从低到高排列，摩擦越低采集面越广、但单条信号越弱；摩擦越高信号越强、但只有少数核心用户会付出。**一个健康的飞轮要在每一层都取信号，不能只押一层。**

| 层 | 信号 | 摩擦 | 喂给哪个引擎 |
|---|---|---|---|
| **L0 隐式** | 选了哪个角色 | 零 | 冷启动分布 |
| | 悬停 / 点开了哪些书 | 零 | 兴趣先验 |
| | **点了 Open Library / 豆瓣跳转** | 零 | **强兴趣信号**——他想去找这本书 |
| **L1 轻标记** | 想读 / 在读 / 读过（书架三态） | 一次点击 | 水平推断 + 已读排除 |
| | Hold 申诉：「我觉得这本没死」 | 一次点击 | 衰变引擎校准 |
| **L2 结构化** | 读完后：对你当前工作有用吗 | 几次点击 | **排环 affinity 的替换燃料** |
| | 「它该在哪个环」——拖到别的环 | 拖拽 | 排环校准 |
| **L3 贡献** | 提名新书 | 高 | 书目库扩充 |
| | 写一句话理由 / 标注衰变信号 | 高 | 内容护城河 |

### PM 判断：信号设计必须问对问题

这是**只有 PM 能定、且定错就前功尽弃**的一件事。

L2 的完读反馈，绝对不能问「这本书打几分」。那是豆瓣评分——全网都有，且有严重的马太效应（名气大的书分高）。PRML 能打 9.5 分，但 90% 的人读不完它，这个分数对「我该不该现在读」毫无信息量。

雷达要的不是「书好不好」，是「**该不该你现在读**」。所以问题必须这么问：

- 不问「打几分」，问 **「你读完了吗」**——完读率骗不了人，它是行为不是态度。
- 不问「推荐吗」，问 **「读完后对你当前的工作有实际帮助吗」**——这才是环的语义。
- 附带一定要记 **「你是什么角色、当时什么目标」**——这一条让信号可以被切片。

收上来的**不是一个平均分，而是一个「角色 × 目标 × 完读率 × 有用度」的矩阵**。这个矩阵有两个价值：它本身就能直接排序（不需要 AI），而它同时正好是排环引擎需要的校准信号——**它直接替换掉 `placement.py` 里那张我手写的 affinity 先验表**。

## 3. 功能清单（每个功能都绑定它产生的信号）

原则：**不做不产生信号的功能。** 每一个新功能都必须回答「它产生什么信号、喂给哪个引擎、转动飞轮的哪一环」，否则就是为功能而功能。

| 功能 | 产生的信号 | 转动飞轮的 | 优先级 |
|---|---|---|---|
| **个人书架**（想读/在读/读过） | L1 书架三态 | 水平推断、已读排除，也是账号存在的最小理由 | P0 |
| **读完反馈**（有用/没用 + 角色目标） | L2 完读矩阵 | 排环 affinity 的替换燃料 | P0 |
| **Hold 申诉** | L1 申诉 | 衰变引擎守门——见下方 PM 判断 | P0 |
| **雷达快照与变迁** | ——（留存钩子） | 个人层兑现「雷达的灵魂是移动」 | P1 |
| **书籍提名** | L3 提名 | 书目库众包扩充，过 ISBN 闸门 | P1 |
| **公开「官方雷达」+ 季度变迁** | ——（传播层） | 第 4 节的冷启动破解 | P1 |
| **一句话理由 / 衰变标注众包** | L3 内容 | 内容护城河 | P2 |

### 三个功能背后的 PM 判断

**书架是账号的最小理由，不要一上来就做注册墙。** 先用浏览器本地存储承接书架，零注册即可用；等用户有了跨设备需求，再引导轻账号。注册摩擦是飞轮第一道漏斗，能省则省。

**Hold 申诉是可信度的守门，不是意见箱。** 产品的全部差异化押在「敢说别读」上。如果某本书的申诉集中爆发，说明衰变规则或人工标注错了——这应当自动触发人工复核队列，而不是沉默。申诉率本身就是衰变引擎的健康度指标（见第 7 节）。这件事不做，Hold 环迟早失信，护城河就塌了。

**雷达快照是留存的核心钩子。** 技术雷达最好看的部分是「移动」。在个人层，用户读完一本书 → 雷达重排 → 他回来看到「我的雷达变了」。这个反馈闭环是让用户第二次、第三次回来的唯一理由，也顺便持续产生 L1/L2 信号。没有它，用户看完一次就再不回来，飞轮转一圈就停。

## 4. 破解冷启动悖论

飞轮需要数据才能转，但零数据时怎么启动？两个不依赖用户数据的引擎：

**第一批燃料靠官方策展，不靠用户。** 这就是为什么阶段一是「纯规则 + 人工策展」——它让产品在**零用户数据**时就有价值。种子书目库和规则衰变引擎是官方先垫进去的燃料，用户来之前飞轮就已经能空转一圈。

**第一批用户靠内容本身的传播性。** Hold 环的「敢说别读」是天然的传播钩子。一张「这 6 本 AI 书别读了，它们讲的范式已经过时」的图，本身就有转发欲——因为它稀缺、有观点、且反直觉（别的书单只会加书，不会劝退）。用内容把第一批用户吸进来，他们产生第一批信号，飞轮才真正开始转。

**顺序不能反**：先官方燃料，再用户数据。指望用户数据来启动，是把飞轮的产物当成飞轮的燃料。

## 5. 飞轮的三个转速

飞轮不是一次建成的，随数据积累提速。**每一档都先跑通再进下一档，不要跳档。**

| 转速 | 机制 | 前提 | AI 参与 |
|---|---|---|---|
| **一档 · 手动** | 用户行为 → 人工看数据 → 手工调 affinity 表和书目库 | 现在就能做 | 无 |
| **二档 · 半自动** | 完读矩阵直接驱动排序（协同过滤式）；申诉自动进复核队列 | 有一定量的 L2 信号 | 无 |
| **三档 · AI** | AI 用真实矩阵做冷启动预测、生成个性化理由、从社区讨论自动发现衰变信号 | 数据足够密 | 有 |

### PM 判断：AI 什么时候进来

**成本不是推迟 AI 的理由**（实测个性化一次调用约 5 分钱，一万张雷达约 559 美元，小到不影响任何架构决策）。真正的理由是：**数据不足时引入 AI，会放大噪声，且无法验证增益。**

AI 的个性化质量由输入数据的质量决定。在没有真实完读矩阵时，AI 的「因人而异」只是把我手写的先验换一种方式重新猜一遍——听起来更高级，实则不可验证。等二档的完读矩阵积累起来，AI 进来做两件它真正不可替代的事：

1. **冷启动预测**——新书还没有任何打分数据时，先给它一个位置。
2. **翻译成人话**——把「角色 × 目标 × 完读率 × 有用度」的矩阵，写成一句能读的推荐理由。

所以「先做飞轮、后上 AI」不是保守，是**给 AI 攒燃料**。

## 6. 数据模型（PM 该定的最小骨架）

飞轮的原料是一条**事件流**。不需要复杂 schema，一张表足够启动：

```
event = {
  actor:      角色 or 匿名 id（本地存储生成，跨设备时升级为账号）
  book:       ISBN（永远锚定真实书目库，见防幻觉铁律）
  action:     view | click | jump | shelve | finish | rate | appeal | nominate
  context:    { role, goal, ring_at_time }   ← 让信号可切片
  value:      因 action 而异（rate 带 useful/not；appeal 带一句话）
  ts:         时间戳
}
```

这条事件流同时是三大引擎的校准输入，也是第 7 节所有指标的来源。**先把这条流接上，比做任何单个功能都重要**——它是飞轮的总管道。

隐私：匿名优先，本地存储先行；任何涉及用户身份的数据，采集前先定用途和留存期，不做「先存着再说」。

## 7. 怎么知道飞轮转起来了

| 指标 | 衡量什么 | 目标方向 |
|---|---|---|
| **北极星：已读书回访更新率** | 用户是否回来更新已读书——说明雷达真在指导他 | ↑ |
| 人均有效信号 / 周 | 飞轮的燃料流量 | ↑ |
| L2 完读矩阵覆盖密度 | 有多少「角色×书」格子有真实数据 | ↑ |
| **Hold 申诉率** | 衰变引擎的健康度——**太高说明判死判错了** | 稳定在低位 |
| 书目库覆盖度 | 提名转化为入库的速度 | 追上领域演进 |

北极星刻意不选「日活」或「停留时长」。雷达不是内容消费产品，它的价值是**指导决策**——用户回来更新已读书，才证明它真的在指导，而不只是被看了一眼。

## 8. 诚实的风险清单

- **信号稀疏**：完读是低频行为（一本书读几周）。矩阵会长期稀疏。缓解：L0/L1 的高频弱信号先撑住排序，L2 慢慢填。不要指望完读矩阵短期内变密。
- **申诉变吵架**：Hold 是有观点的判断，必然有人不同意。申诉要能承接分歧而不动摇立场——设计成「触发复核」，不是「谁声大听谁的」。
- **提名灌水**：众包入口会引来营销和自荐。ISBN 闸门挡掉假书，但挡不掉「真书但不该进」。需要人工审核关，不能全自动。
- **冷启动传播依赖单点**：第一批用户全靠 Hold 的传播性。如果那张「别读」的图没传起来，飞轮启动不了。这是最大的单点风险，值得在内容和分发上单独投入。

---

## 9. 建议的下一步

按飞轮供能优先级，不按功能大小：

1. **接事件流**（第 6 节）——飞轮的总管道，先于一切单个功能。
2. **书架 + 读完反馈**（P0）——L1/L2 信号的最小采集面。
3. **Hold 申诉**（P0）——守住护城河。
4. **雷达快照**（P1）——留存钩子，让用户回来。
5. 数据攒够后，才谈二档（协同过滤排序）和三档（AI 进场）。

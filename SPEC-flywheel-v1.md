## 一、事件流：schema 与埋点

事件流是飞轮的总管道。三大引擎（排环 affinity、衰变规则、书目库）都从它取校准输入，第 7 节所有指标也从它派生。**先接通这条管道，比做任何单个功能都重要。**

### 1.1 身份模型（actor）

| 阶段 | 机制 | 摩擦 | 代价 |
|---|---|---|---|
| 起步 | 匿名 `uuid`，存浏览器本地 | 零注册 | 跨设备/清缓存即丢失 |
| 升级 | 用户自愿绑轻账号（邮箱或 OAuth），把匿名 `uuid` 归并进账号 | 一次登录 | 需后端账号体系 |

**书架的个人体验（本地）和飞轮的数据采集（中心）是两件事，可以解耦。** 书架可以纯本地存、零注册即用；同时把匿名事件异步打到后端做飞轮燃料。这样既不加注册摩擦，又能接上飞轮——这是本规格推荐的路线。

### 1.2 事件结构

```jsonc
{
  "actor":   "anon_9f3c…",          // 匿名 uuid，或升级后的账号 id
  "action":  "rate",                // 见下方枚举
  "book":    "9780521642989",       // ISBN，永远锚定真实书目库；nominate 时为空
  "context": {
    "role":          "researcher",  // 当时选的角色
    "goal":          "读懂并复现前沿论文",
    "ring_at_time":  "adopt",       // 事件发生时这本书在哪个环
    "radar_version": "2026Q3",      // 雷达快照版本，用于关联「移动」
    "session_id":    "s_7a1b…"
  },
  "value":   { "finished": "yes", "useful": "high" },  // 因 action 而异，见枚举
  "ts":      "2026-07-16T10:22:04Z"
}
```

`context` 是让信号可被切片的关键——同一本书，研究员觉得有用、产品经理觉得没用，只有带上 `role`/`goal` 才分得开。

### 1.3 action 枚举（这是埋点的权威清单）

| action | 触发点 | value | 信号层 | 喂给 |
|---|---|---|---|---|
| `session_start` | 进入首页 | — | — | 漏斗基数 |
| `role_select` | 选角色 | `{role}` | L0 | 冷启动分布 |
| `open` | 点开某本书的详情 | — | L0 | 兴趣先验 |
| `jump` | 点 Open Library / 豆瓣外链 | `{target}` | L0 强 | 强兴趣——他要去找书 |
| `path_click` | 点阅读路径某项 | — | L0 | 路径有效性 |
| `shelve` | 加入书架 / 改状态 | `{state: want\|reading\|read\|removed}` | L1 | 已读排除、水平推断 |
| `rate` | 读完反馈 | `{finished: yes\|partial\|dropped, useful: high\|some\|none, note?}` | **L2** | **排环 affinity 替换燃料** |
| `appeal` | Hold 申诉 | `{note}` | L1 | 衰变引擎校准 |
| `nominate` | 提名新书 | `{title, author}` | L3 | 书目库扩充（过 ISBN 闸门） |

**刻意不记的**：原始 `hover`。它量极大、信号极弱，`open`（点开详情）已经足够表达兴趣。记 hover 只会淹没有用信号、抬高存储成本。这是一个明确的取舍，不是遗漏。

### 1.4 存储形态

- 事件流 **append-only**（只追加，不改不删）——这是飞轮的原始账本，也是日后一切分析和回溯的依据。
- 个人书架是事件流的**派生视图**（可另缓存一份当前态以便快速渲染），不是独立的事实来源。派生态出错时，能从事件流重建。
- 一张表即可启动，无需复杂 schema。

### 1.5 隐私（采集前先定，不「先存着再说」）

- 匿名优先。`actor` 是随机 uuid，不含身份信息。
- 升级账号时才采集邮箱，且仅用于登录与（若用户开启的）读完提醒，用途单一。
- 事件流定明确留存期；`note` 类自由文本（申诉、提名）走内容审核，防止 PII 与灌水。

## 二、书架 + 读完反馈：用户旅程

书架是账号存在的最小理由，也是 L1/L2 信号的主采集面。读完反馈是**唯一能产出「该不该你现在读」这个核心信号的动作**，是整条旅程的重心。

### 2.1 一本书在用户视角的状态机

```
   未标记 ──想读──▶ 想读 ──开读──▶ 在读 ──读完──▶ 读过
     │                                            │
     └──────────── 直接标「读过」─────────────────┘
                                                  │
                              读过后追加：完读确认 + 有用度  →  L2 燃料
```

每一次状态转移都发一条 `shelve` 事件；读完反馈发一条 `rate` 事件。

### 2.2 三条关键旅程

**① 发现并收藏**
雷达上点开一本书 → 详情卡出现「想读」按钮 → 点击 → 落进书架「想读」。
产生：`open` + `shelve(want)`。零摩擦，是最高频的收藏动作。

**② 开始读**
书架里把「想读」推进到「在读」。产生：`shelve(reading)`。
这一步可选——很多人读完才想起标记，所以不强求经过「在读」。

**③ 读完反馈（旅程重心）**
标记「读过」时，弹出最小反馈：

- **完读确认**：读完了吗？　读完 / 读了部分 / 弃了
- **有用度**：对你当时的目标有帮助吗？　很有用 / 一般 / 没用
- （可选）一句话：哪里有用 / 哪里没用

产生：`shelve(read)` + `rate({finished, useful, note?})`。

这里有一个硬权衡：**问得越多信号越全，但完成率越低。** 反馈问卷问几个字段，直接决定 L2 矩阵能不能填起来——是本规格最需要决策的一点（见待确认 Q3）。

### 2.3 读完后，雷达怎么变

这是留存的核心闭环——**用户读完一本书，回来看到「我的雷达变了」。**

- **已读排除**：读过的书不再作为推荐候选（`placement.py` 已实现 `read` 排除）。
- **水平推断**：读过硬核书 → 提示系统「他能啃硬的」→ 后续排序对该角色放宽难度约束。
- **重排 + 快照 diff**：重新排环，与上一版快照做 diff，标出移动的书——这就是个人层的「雷达的灵魂是移动」。

**已读书在雷达上是消失还是留下**，是一个 UX 兼留存决策（见待确认 Q4）。留下并点亮，能给用户「我的雷达点亮了几颗」的成就轨迹，是让人回来的钩子；消失则更干净。

### 2.4 把用户拉回来填读完反馈

读完是低频行为——用户读完往往已离开产品几周。L2 燃料的采集，卡在「怎么把人拉回来填」。三档强度：

1. **不主动提醒**：靠自然回访（看自己的雷达）承接。零打扰，但采集率低。
2. **站内待办**：下次进来时，「在读」的书顶部提示「读完了吗？」。中等。
3. **邮件提醒**：需要邮箱（即需要升级账号），能主动召回，但有打扰成本。

强度与账号策略耦合——无邮箱就没有邮件提醒。见待确认 Q2。

## 三、需要你确认的决策清单

以下五项会改变后续的范围、工期与数据能力，需要裁决。前四项已用交互问题摆出，第五项在此备案。

| # | 决策 | 影响 |
|---|---|---|
| Q1 | 本期范围：只做本地个人体验，还是同时接通后端飞轮数据管道 | 决定这一期是「demo」还是「飞轮地基」 |
| Q2 | 账号策略：纯匿名 / 匿名+可选登录 / 强制登录 | 漏斗摩擦 vs 数据能力（含能否邮件召回） |
| Q3 | 读完反馈的必答字段数：1 步 / 2 步 / 3 步 | 完成率 vs L2 信号完整度 |
| Q4 | 已读书在雷达上的呈现：留下点亮 / 读过即移除 | 留存钩子 vs 视觉干净 |
| Q5 | 拉回来填反馈的强度：不提醒 / 站内待办 / 邮件召回 | 与 Q2 耦合；决定 L2 采集率 |

**待你在这些决策上拍板后，本文档会更新为可执行的一期实施清单。**
